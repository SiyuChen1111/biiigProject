## 2. 这个项目整体在做什么

这个项目是一个 **EEG / CPP（Centroparietal Positivity，中央顶叶正波）研究工程仓库**。它的核心任务可以用一句话概括：

> 用可复现的数据处理和建模流程，研究 EEG 中与 CPP 相关的信号结构，并尝试做分类与生成。

更具体地说，这个项目目前主要分成三条线：

1. **Phase 1：判别式分类**  
   从 EEG 中提取 CPP 相关特征，构造高/低 CPP-like 标签，再训练 EEGNet 或基线模型做分类。

2. **Phase 2：外部标签探索**  
   不再用“CPP-like 高低”作为标签，而改用任务标签或行为标签，比如 `cue_dimensionality` 或反应时快慢，看 EEG 是否还能稳定跨被试分类。

3. **Conditional generation：条件生成**  
   用条件 VAE 生成带有 CPP-like 条件标签的 EEG 片段，探索模型能不能不仅分类，还能生成合理的 EEG 波形。

从仓库结构上看，这个项目已经从“笔记本驱动”的早期状态，逐渐过渡到“模块化 Python 工程”的结构。

## 3. 根目录下每个主要位置是什么意思

### `README.md`

这是整个项目的总入口。它说明了：

- 项目采用一种受 TIER 协议启发的可复现目录结构；
- 当前主要研究对象是 Kosciessa 的 `task-dynamic` EEG 数据；
- `src/` 是可复用代码，`scripts/` 是运行入口，`docs/` 是文档，`outputs/` 是保留结果；
- 当前项目状态：Phase 1 已完成并验证，Phase 2 是探索性尝试，条件生成分支也已经实现；
- 下一步更推荐聚焦 ROI（`CPz/CP1/CP2`）的生成，而不是一开始就全头皮大规模生成。

如果你第一次进入这个仓库，应该先看这个文件。

### `.gitignore`

这是 Git 忽略规则文件。它告诉 Git：

- `data/raw/` 和 `1_Data/` 里的原始数据不要提交；
- Python 运行产生的缓存不要提交；
- macOS 产生的系统文件不要提交；
- `low-level_1.pdf` 这个 PDF 也被忽略。

它的作用是保证仓库只保存“代码、文档、关键结果”，而不是把大体积原始数据和临时文件一起版本化。

### `requirements.txt`

这是 Python 依赖清单。里面列出的库说明了这个项目主要依赖：

- `mne`：处理 EEG/ERP 数据；
- `numpy`、`pandas`：数值和表格处理；
- `matplotlib`：画图；
- `scikit-learn`：传统机器学习；
- `torch`：深度学习；
- `PyPDF2`、`pdfplumber`、`PyMuPDF`：处理 PDF。

换句话说，这个项目是一个典型的“EEG 信号处理 + 机器学习/深度学习 + 文档整理”的 Python 研究工程。

### `1_Data/`

这是旧式原始数据目录。按照 `README.md` 的说法，它是为了向后兼容而保留的。当前代码更偏好 `data/raw/...` 结构，但如果那个路径不存在，就会回退到这里找数据。

这里的重点不是文本文件本身，而是它代表了项目目前还处在“过渡式重构”阶段：新结构和旧结构都还在兼容。

### `2_Analysis/`

这是旧分析流程保留下来的目录。里面有旧版 CPP 管线的代码和说明文档。它对理解项目历史特别重要，因为可以看到：

- 最初逻辑是怎么从 notebook 风格逐步演化出来的；
- 旧版标签定义和新版 `src/ + scripts/` 结构之间有哪些对应关系。

### `docs/`

这是文档区。它分成：

- `notes/`：早期想法和项目思路；
- `protocol/`：更正式的蓝图、总结、提案、结果索引。

如果你想理解“为什么要做这个项目”“这个项目的科学问题是什么”，这里比代码更重要。

### `outputs/`

这是保留实验结果的目录。这里放的是：

它不是源代码，而是项目当前结论的“证据库”。

这是项目的可执行入口层。通常你真正运行实验时，不是直接导入 `src/`，而是运行这里的脚本。

## 5. `src/` 目录里的文件逐个解释

## `src/`

### `src/__init__.py`

这是一个空文件。它本身不承载逻辑，作用是把 `src/` 明确标记成 Python 包目录，方便通过 `src.xxx` 的方式导入子模块。

## `src/data/`

### `src/data/__init__.py`

也是空文件，作用同上，用来声明 `src.data` 是一个可导入的子包。

### `src/data/data_kosciessa.py`

这是新版本数据层的核心文件之一，负责 **定位数据集、列出被试、读取表格、构造 trial table**。它主要做了几件事：

- 定义默认数据路径和旧路径；
- 自动判断优先使用 `data/raw/...` 还是回退到 `1_Data/...`；
- 定义 `SubjectPaths` 这个数据结构，把一个被试的 behavior / EEG / events / channels 文件路径打包起来；
- 提供读取 behavior、events、channels 表格的函数；
- 提供 `build_subject_trial_table()`，把行为表和事件表整理成一个更适合后续建模的 trial 级表；
- 检查行为里的响应样本点是否和事件标记大致对齐。

简单说，这个文件负责把“原始数据文件”整理成“后续代码能直接用的试次表”。

## `src/preprocessing/`

### `src/preprocessing/__init__.py`

空文件，用来声明 `src.preprocessing` 是包。

### `src/preprocessing/epoching.py`

这是预处理和 epoch 提取的核心文件。它定义了：

- Phase 1 固定使用的通道数、时间窗口、基线窗口、目标采样率；
- `EpochedSubjectData` 这个数据结构，用来存每个被试的 epoch 数据；
- `load_raw_eeg()`：读取 BrainVision EEG，保留 EEG 通道，做平均参考和 1–30 Hz 滤波；
- `_extract_epoch()`：按响应点切出一个时间窗；
- `_baseline_correct()`：做基线校正；
- `_resample_epoch()`：重采样到目标频率；
- `extract_response_locked_epochs()`：把上述步骤串起来，输出每个被试的 response-locked epoch。

这是“把连续 EEG 变成模型输入片段”的关键文件。

## `src/features/`

### `src/features/__init__.py`

空文件，用来声明 `src.features` 是包。

### `src/features/labels_cpp.py`

这是新版本 CPP 特征和标签定义文件，负责把 epoch 转成 **CPP 相关特征、分数和标签**。它的主要逻辑是：

- 固定 ROI 通道：`CPz`、`CP1`、`CP2`；
- 固定三个时间窗：`AMS`、`PAMS`、`SLPS`；
- 先在 ROI 内求平均信号；
- 再算三个特征：平均幅值、峰值、斜率；
- 用训练集均值和方差做 z-score；
- 把三个 z-score 加总成 `cpp_score`；
- 用 25% / 75% 分位阈值把 score 切成低/高两类，中间区域丢弃。

换句话说，它把“什么叫高 CPP-like，什么叫低 CPP-like”这件事编码成了一个清楚的规则。

## `src/evaluation/`

### `src/evaluation/__init__.py`

空文件，用来声明 `src.evaluation` 是包。

### `src/evaluation/evaluate_phase1.py`

这是评估和画图模块，负责：

- 计算二分类指标，如 balanced accuracy、precision、recall、ROC-AUC；
- 统计混淆矩阵中的 `tn/fp/fn/tp`；
- 汇总不同 fold 的指标均值和标准差；
- 画 fold 层面的指标折线图；
- 画按被试拆分的 balanced accuracy 柱状图；
- 画 ROI ERP 正负类对比图。

所以它的角色不是训练模型，而是把模型输出整理成可解释的结果和图表。

## `src/models/`

### `src/models/__init__.py`

空文件，用来声明 `src.models` 是包。

### `src/models/EEGModels_PyTorch.py`

这是仓库里最重要的模型定义文件之一。它是一个较完整的 PyTorch EEG 模型库，核心内容包括：

- `EEGNet` 的 PyTorch 实现；
- `DepthwiseConv2d`、`SeparableConv2d` 等专用卷积层；
- Max-norm 约束；
- 一些为 EEGNet 配套的工具接口；
- 支持提取特征、替换分类头、输出概率等迁移学习友好的方法。

它并不只服务当前这个项目，它更像是这个项目里可复用的“EEG 模型资产”。

### `src/models/ERP_PyTorch.py`

这是一个示例性质的脚本/模型使用样例，演示如何用 EEGNet 在 MNE 的示例 ERP 数据集上做分类。它主要包含：

- 下载并读取 MNE 示例数据；
- epoch 数据；
- 训练 EEGNet；
- 同时对比 PyRiemann 方法；
- 输出准确率和混淆矩阵。

它更像“教学样例”或“验证 EEGNet 基本工作方式的示范文件”，不是当前项目主实验入口。

### `src/models/conditional_eeg_vae.py`

这是条件生成分支的核心模型文件。它定义了 `ConditionalEEGVAE`，也就是条件 EEG 变分自编码器。它的关键点是：

- 编码器骨干直接借用了 `EEGNet` 的特征提取能力；
- 输入除了 EEG 以外，还结合了类别标签条件；
- 潜变量用 `mu / logvar` 参数化；
- 解码器用一维反卷积逐步生成 EEG；
- 训练损失不只有重建 loss 和 KL loss，还支持：
  - feature consistency loss；
  - smoothness loss；
  - ROI reconstruction loss；
  - label loss。

也就是说，这个模型不是简单地“重建 EEG”，而是在努力让生成结果保留 CPP 相关结构和标签信息。

## 6. `scripts/` 目录里的文件逐个解释

### `scripts/run_master.py`

这是一个总控入口脚本。它通过命令行参数把任务转发到不同实验脚本，比如：

- `phase1`
- `phase1-baseline`
- `phase2-cue`
- `phase2-rt`

它的价值是：给用户一个统一入口，而不用记住每个脚本的完整路径。

### `scripts/train_phase1_eegnet.py`

这是 Phase 1 的主脚本，也是最核心的训练入口之一。它做的事情很多：

- 读取被试 trial table；
- 提取 response-locked EEG epochs；
- 计算 CPP 特征；
- 根据训练集拟合标签变换规则；
- 做按被试的 5 折交叉验证；
- 每一折再做训练/验证集划分；
- 训练 EEGNet；
- 用验证集挑最优 checkpoint；
- 在测试被试上评估；
- 保存 fold、summary、subject 三级指标；
- 画 ROI ERP 图和其他汇总图。

它基本上就是“Phase 1 全流程的主驱动器”。

### `scripts/baseline_phase1_cpp_features.py`

这是 Phase 1 的传统机器学习基线脚本。和 EEGNet 脚本相比，它不直接吃原始 epoch，而是只用：

- `ams`
- `pams`
- `slps`

这三个手工特征，配合 `StandardScaler + LogisticRegression` 去分类。

它的科学意义非常大，因为它回答的是：

> 深度模型到底有没有明显超越简单手工特征？

在这个项目里，这个问题恰恰是当前结论的核心之一。

### `scripts/train_phase2_cue_dimensionality.py`

这是 Phase 2 的一个探索脚本，目标不是预测 CPP-like 标签，而是预测外部任务标签 `cue_dimensionality` 的极端值。它整体流程和 Phase 1 EEGNet 很像，但标签来源改成了行为/任务变量。

它的作用是测试：

- 如果不用“从 EEG 本身构造出的 CPP-like 标签”，
- 而改用外部任务标签，
- EEG 是否还能稳定学到跨被试可泛化信号。

当前结果显示，这条线还没有明显跑通。

### `scripts/train_phase2_rt_fastslow.py`

这和上一个脚本类似，但它把标签定义成每个被试内部的 **反应时快 vs 慢**。也就是说：

- 先算每个被试自己的 RT 中位数；
- 比中位数快的标 0，慢的标 1；
- 再用 EEGNet 做跨被试分类。

它也是一种探索性外部标签实验，用来检验当前 EEG 管线在行为标签上的泛化能力。

### `scripts/train_phase1_conditional_vae.py`

这是条件生成分支的主训练脚本。它的工作包括：

- 加载被试数据和 CPP 特征；
- 用训练集构造 Phase 1 的高/低 CPP-like 标签；
- 把 epoch 转成 VAE 输入；
- 训练 `ConditionalEEGVAE`；
- 在验证集上选择最优模型；
- 在测试集统计不同 loss；
- 为每折生成不同类别条件下的样本；
- 保存 `fold_losses.json` 和 `summary_losses.json`。

这是整个生成建模流程的主要入口。

### `scripts/analyze_conditional_vae_samples.py`

这是生成结果的分析脚本。它的重点不是训练，而是：

- 加载真实数据；
- 对真实测试数据重新构造高/低 CPP-like 标签；
- 读取 VAE 生成出来的样本数组；
- 把真实和生成的 ROI ERP 波形做比较；
- 计算真实正负条件和生成正负条件的均值差距；
- 输出比较图和 `fold_generation_summary.json`。

可以把它理解为“生成结果质检脚本”。

### `2_Analysis/CPP_PIPELINE_README.md`

- 输出会生成什么；
- 限制在哪里。

这个文件很像旧时代管线的“使用说明书 + 技术注释”。它对理解历史设计很重要。

### `2_Analysis/cpp_paths.py`

这是旧版管线的路径配置文件。它集中定义：

- 项目根目录；
- 数据集根目录；
- 输出目录；
- CPP epoch 输出目录；
- ANN 输出目录。

作用是避免在多个文件里硬编码路径。

### `2_Analysis/cpp_epoch_pipeline.py`

这是旧版 epoch 导出流程的主文件。它大体负责：

- 找到所有被试；
- 读取 EEG 和行为文件；
- 清理行为表；
- 推导 response sample；
- 建立 response-locked epochs；
- 提取 CPP 相关通道；
- 计算波形特征；
- 生成 metadata；
- 导出 epoch tensor、CPP waveform、score、label、times 等文件。

这个文件的角色和新架构中的 `src/data + src/preprocessing + src/features` 加在一起有点类似，只不过旧版还没有拆得那么清楚。

### `2_Analysis/cpp_labeling.py`

这是旧版 CPP 标签规则文件。它做的事情和新版 `src/features/labels_cpp.py` 很像，但定义方式略有不同。主要逻辑包括：

- 先对多个特征做被试内 z-score；
- 再加权形成 `cpp_score`；
- 用一个 morphology gate 判断形态是否符合“上升、接近响应时正幅、响应后下降”；
- 再按被试中位数阈值分配 `cpp_label`。

你可以把它看成“旧版 CPP 判定规则的显式实现”。

### `2_Analysis/cpp_ann.py`

这是旧版 ANN 训练脚本。它会：

- 读取导出的 epoch 数据；
- 展平 tensor；
- 按被试划分 train/val/test；
- 用 `StandardScaler + PCA + MLPClassifier` 训练 ANN；
- 输出指标和预测。

它的意义在于，旧版管线并不是只有特征提取，还已经包含了一个相对完整的分类基线流程。

### `2_Analysis/1_eeg_data_preprocess.ipynb`

这是旧版预处理 notebook。从 `CPP_PIPELINE_README.md` 的说明来看，它最早用于：

- 读取原始 EEG 和行为数据；
- 做数据清洗；
- 产出适合后续分析的数据结构。

它现在的主要价值已经不是“当前主入口”，而是保留项目早期预处理逻辑，方便回看历史设计。

### `2_Analysis/2_epoch_data.ipynb`

这是旧版 epoch 和特征提取 notebook。它主要承担的是：

- 建立 response-locked epochs；
- 计算 trial 级 CPP 相关特征；
- 为后面的标签和训练做准备。

它更像新模块化实现诞生前的“原型实验笔记本”。

### `2_Analysis/2_cpp_epoch_labeling_and_export.ipynb`

这是旧版 CPP epoch 导出 notebook。它相当于给旧模块化代码做了一个 notebook 包装层，便于：

- 运行 epoch 导出；
- 查看标签分布；
- 快速检查输出张量形状和导出文件。

如果把 `cpp_epoch_pipeline.py` 看成发动机，这个 notebook 更像是仪表盘。

### `2_Analysis/3_ann_training.ipynb`

这是旧版 ANN 训练 notebook，对应 `cpp_ann.py` 的 notebook 入口。它的定位是：

- 用 notebook 方式运行旧 ANN 训练；
- 查看训练结果和预测输出；
- 降低对纯命令行的依赖。

它属于旧流程里“更方便交互式使用”的那一层。

## 8. `outputs/` 目录里的文本结果文件逐个解释

先说明一下：`outputs/` 里除了 JSON 以外，还会有 PNG 图、Numpy 数组等非文本文件。这些非文本文件我这里不逐一展开，但会说明它们所在目录的意义。

## `outputs/phase1_midrun/`

### `outputs/phase1_midrun/fold_metrics.json`

这是 Phase 1 EEGNet 主实验的每折指标。它记录每一折的：

- balanced accuracy
- precision
- recall
- tn / fp / fn / tp
- roc_auc
- test 被试数和 trial 数

从内容上看，这份文件反映出 EEGNet 在 Phase 1 上表现非常强，多数折都很高。

### `outputs/phase1_midrun/summary_metrics.json`

这是把上面 5 折指标做均值和标准差汇总后的结果。它给出的是这个实验的一眼总览。

### `outputs/phase1_midrun/subject_metrics.json`

这是按被试拆开的指标结果。它的作用是帮助检查：

- 模型是不是对某些被试特别好、对某些被试特别差；
- 是否存在个体差异较大的情况。

## `outputs/phase1_baseline_midrun/`

### `outputs/phase1_baseline_midrun/fold_metrics.json`

这是 Phase 1 手工特征逻辑回归基线的一组未缩放结果。它显示这个版本的基线表现并不好，很多折的 balanced accuracy 接近 0.5。

### `outputs/phase1_baseline_midrun/summary_metrics.json`

这是未缩放基线的汇总结果。说明这个版本的基线总体不可靠。

### `outputs/phase1_baseline_midrun/subject_metrics.json`

这是未缩放基线按被试拆开的结果。能看出许多被试上预测几乎塌成单类。

## `outputs/phase1_baseline_midrun_scaled/`

### `outputs/phase1_baseline_midrun_scaled/fold_metrics.json`

这是缩放后的手工特征基线结果。和上面的未缩放版本相比，这里几乎是“完美分类”。

### `outputs/phase1_baseline_midrun_scaled/summary_metrics.json`

这是缩放基线的汇总，所有指标都是 1.0。它非常强烈地提醒我们：

- 手工特征本身就非常强；
- 不能简单因为 EEGNet 准确率高，就说深度模型学到了更深层的东西。

### `outputs/phase1_baseline_midrun_scaled/subject_metrics.json`

这是缩放基线按被试拆开的完美结果。它进一步支持上面的结论：当前 Phase 1 任务本质上和手工定义的 CPP-like score 非常接近。

## `outputs/phase2_quickcheck/`

### `outputs/phase2_quickcheck/fold_metrics.json`

这是 Phase 2 `cue_dimensionality` 快速检查实验的每折结果。整体看 balanced accuracy 约在 0.45–0.52 左右，接近随机。

### `outputs/phase2_quickcheck/summary_metrics.json`

这是 Phase 2 quick check 的汇总。它说明目前这条外部标签任务还没有形成可靠信号。

### `outputs/phase2_quickcheck/subject_metrics.json`

这是 Phase 2 quick check 的按被试结果。它说明某些被试上会偏向某一类，整体不稳定。

## `outputs/phase1_conditional_vae_auxquick/`

### `outputs/phase1_conditional_vae_auxquick/fold_losses.json`

这是条件 VAE 每折的 loss 记录。它不仅有总 loss，还有组成部分：

- recon_loss
- kl_loss
- feature_loss
- smoothness_loss
- roi_recon_loss
- label_loss

这说明生成模型训练是“多目标约束”的，不只是做普通重建。

### `outputs/phase1_conditional_vae_auxquick/summary_losses.json`

这是上面各类 loss 的总体均值和标准差。它用来概括当前生成训练的整体状态。

## `outputs/phase1_conditional_vae_auxquick_analysis/`

### `outputs/phase1_conditional_vae_auxquick_analysis/fold_generation_summary.json`

这是生成样本分析结果的数值摘要。它主要比较：

- 真实高/低条件间的 ROI 平均差；
- 生成高/低条件间的 ROI 平均差；
- 两者差距的绝对误差。

这个文件反映的是：生成模型虽然能生成样本，但真实条件差异与生成条件差异之间还没有达到很理想的一致性。

## `2_Analysis/outputs/cpp_epochs/`

这是旧版 CPP epoch 导出流程生成的数据区，主要保存“供旧 ANN 或后续分析使用的中间产物”。

### `2_Analysis/outputs/cpp_epochs/epoch_metadata.csv`

这是旧版 epoch 数据的元数据总表。它非常大，按行记录每一个 trial / epoch 的信息，包括：

- 行为变量；
- 被试信息；
- CPP 特征；
- z-score 后特征；
- `cpp_score`、`cpp_label`、阈值等。

它的作用相当于“旧版分析流程的总索引表”。你如果想知道某个 epoch 的标签是怎么来的，这个表非常关键。

### `2_Analysis/outputs/cpp_epochs/epoch_tensor.npy`

这是旧版流程导出的 epoch 张量数组文件。虽然它不是文本文件，但它和 `epoch_metadata.csv` 是成套出现的：前者存实际 EEG 数值，后者存解释这些数值所需的表格信息。

### `2_Analysis/outputs/cpp_epochs/epoch_cpp_waveform.npy`

这是旧版流程中导出的 CPP 波形数组，通常用于分析或可视化 ROI 波形。

### `2_Analysis/outputs/cpp_epochs/epoch_score.npy`

这是旧版流程导出的每个 epoch 的 CPP 分数数组文件，对应 metadata 里的 `cpp_score` 列。

### `2_Analysis/outputs/cpp_epochs/epoch_label.npy`

这是旧版流程导出的标签数组文件，对应 metadata 里的 `cpp_label` 列。

### `2_Analysis/outputs/cpp_epochs/epoch_times.npy`

这是 epoch 时间轴数组，说明每个时间点相对响应锁定点的位置。

## `2_Analysis/outputs/ann/`

这是旧版 ANN 分类流程的结果区。

### `2_Analysis/outputs/ann/metrics.csv`

这是旧 ANN 模型的结果汇总表，内容比较简洁，按 `val` 和 `test` 两个 split 记录：

- accuracy
- balanced_accuracy
- f1
- roc_auc
- n_samples

它适合用来快速看旧 ANN 流程整体表现。

### `2_Analysis/outputs/ann/predictions.csv`

这是旧 ANN 的逐样本预测表。它比 `metrics.csv` 细得多，既保留了：

- 原始行为与 CPP 元数据；
- 真值标签；
- 预测标签；
- 预测概率；
- 所属 split。

也就是说，它是“结果分析明细表”。如果你想检查模型在哪些 trial 上分错，应该看这个文件。

### `2_Analysis/outputs/ann/mlp_classifier.joblib`

这是旧 ANN 训练后保存的模型文件，不是文本文件，但它和上面的 `metrics.csv`、`predictions.csv` 一起构成旧版 ANN 结果包。

## 9. 一些空文件也值得说明

仓库里有多个空的 `__init__.py` 文件，例如：

- `src/__init__.py`
- `src/data/__init__.py`
- `src/preprocessing/__init__.py`
- `src/features/__init__.py`
- `src/models/__init__.py`
- `src/evaluation/__init__.py`

这些文件虽然“没有内容”，但并不是没用。它们的作用是：

- 把目录标记为 Python 包；
- 让模块之间可以稳定用 `src.xxx` 导入；
- 使整个项目更接近标准 Python 工程结构。

所以在“工程结构”层面，它们是有意义的。

## 10. 这个仓库的结构关系可以怎么理解

如果把整个项目当成一条研究流水线，可以这样理解：

1. **想法层**：`docs/notes/*`  
   解释为什么想研究 CPP 检测与特异性。

2. **协议/设计层**：`docs/protocol/*`  
   把问题正式化，写清楚实验蓝图、当前结论和后续方向。

3. **核心代码层**：`src/*`  
   负责数据、预处理、特征、模型、评估，是新架构的中心。

4. **运行入口层**：`scripts/*`  
   把 `src/` 中的模块组织成可直接运行的实验流程。

5. **历史管线层**：`2_Analysis/*`  
   保留旧版 notebook / 旧版模块化流程，方便回看和比较。

6. **证据层**：`outputs/*`  
   保存已经跑出来的结果，用于支持当前结论。
